Nibble

Go-live guide

How to run the files in VS Code, publish the site on GitHub, and add your market data after launch. Written for a non-developer — follow it top to bottom.

1What you're publishing

Nibble is the front-end of your food-stall app — the homepage, browsing, stall pages, profiles and the admin CMS. It runs entirely in the web browser.

  • For a quick pilot or preview, you can host it for free as a static site on GitHub Pages (this guide).
  • For the full company launch — Google sign-in, one shared database everyone sees, photo uploads visible to all — you'll need the backend from your original brief (Supabase). That's covered in step 7.

2What you'll need (once)

  • A computer running Chrome or Edge.
  • Visual Studio Code — free, from code.visualstudio.com.
  • Git — free, from git-scm.com (accept the defaults during install).
  • A free GitHub account — github.com.

3Get the files onto your computer

  1. Download this project as a .zip (use the download option in the toolbar, or ask me to hand you a download).
  2. Unzip it to a folder you'll remember, e.g. Documents/nibble.
  3. Open VS Code → File → Open Folder… → choose that folder.
  4. You should see Nibble.dc.html, StallCard.dc.html, image-slot.js, support.js and a _ds folder.

4Preview it on your machine

  1. In VS Code, open the Extensions panel (the squares icon on the left), search “Live Server”, and click Install.
  2. Right-click Nibble.dc.html in the file list → “Open with Live Server”.
  3. Your browser opens the working app at an address like http://127.0.0.1:5500/Nibble.dc.html.
Why not just double-click the file?

Always open it through Live Server. The app loads other files (the design system, components), and browsers block those when a page is opened straight from disk — so double-clicking shows a blank page.

5Publish it on GitHub Pages

  1. On github.com, click New to create a repository. Name it nibble, keep it Public, and click Create repository.
  2. Back in VS Code, open Terminal → New Terminal and run these lines one at a time (replace YOUR-USERNAME):
git init
git add .
git commit -m "First version of Nibble"
git branch -M main
git remote add origin https://github.com/YOUR-USERNAME/nibble.git
git push -u origin main
  1. In the repository on GitHub, go to Settings → Pages.
  2. Under Source, choose “Deploy from a branch”, set branch to main and folder to / (root), then Save.
  3. Wait about a minute. Your site is live at:
    https://YOUR-USERNAME.github.io/nibble/Nibble.dc.html
Two handy tips

• Rename Nibble.dc.html to index.html and the address shortens to …github.io/nibble/.

• To update the live site later, just repeat: git add . → git commit -m "update" → git push. Pages redeploys on its own.

6Managing your market data

This build is pre-loaded with the 24 White Cross Street Market stalls and their photos, taken from your spreadsheet. Ratings, reviews and visits all start empty — they fill in as the office uses the app. You're the only account (Daisy Jones, admin). Manage everything from the Admin area:

  • Market stalls → + Add stall — name, category, food type, and tags (pick dietary presets like Vegan or Halal, or type your own); then Edit to drop in a photo.
  • Food items → + Add food item — dishes and prices per stall.
  • Featured stall — choose the Stall of the Week and its reason badge.
  • Settings → People & access — add colleagues by email and grant admin access.
Important — read before you rely on this

On a static GitHub Pages site there is no shared database. Anything an admin adds is saved only in that person's browser — other visitors won't see it, and it's lost if the browser data is cleared. Photo uploads behave the same way.

So GitHub Pages is perfect for a demo or single-machine pilot, but not for a real shared launch across the office. For that, use the backend below.

7The full production version (recommended for a real launch)

Your original brief already specifies the right architecture. The screens in this prototype stay the same — they sit on top of a shared backend:

  • Supabase Auth with Google, restricted to @selinafinance.co.uk — the “no anonymous, company-only” sign-in you asked for (step-by-step in section 10).
  • Supabase Database — one shared Postgres database, so every employee sees the same stalls, ratings and visit counts.
  • Supabase Storage — stall photos uploaded by admins are stored centrally and visible to everyone.
  • Row-Level Security — employees can browse, review and upvote; only admins can edit stalls, moderate reviews and open the CMS.

Deploying that means creating a Supabase project and hosting the app on a platform like Vercel or Netlify (a similar “push to Git” flow). It's more setup than GitHub Pages, but it's the version that supports real logins, shared data and uploads. Happy to scope or build this as the next step.

8Republishing after a change

Whenever you (or I) change the files — new copy, a fix, a design tweak — the live site doesn't update on its own. Push the change and GitHub Pages redeploys within a minute. Do this in the VS Code Terminal:

  1. Make sure you're in the project folder (the terminal prompt shows it). If not: cd Documents/nibble.
  2. Run these three lines, one at a time:
git add .
git commit -m "Describe what changed"
git push
  1. Wait ~1 minute, then hard-refresh the live page (Ctrl/Cmd + Shift + R) to see the update.
If I hand you a new version

Download the updated files, drop them into the same folder (replacing the old ones — say yes to overwrite), then run the three lines above. That's the whole loop.

9Switch the weekly email to a Slack alert

Instead of a Monday email to the whole workspace, Nibble can post to a public Slack channel — e.g. #nibble. Slack does this with an Incoming Webhook (a private URL you post messages to). Set it up once:

  1. In Slack, create the public channel you want alerts in (e.g. #nibble).
  2. Go to api.slack.com/apps → Create New App → From scratch. Name it “Nibble”, pick your Selina workspace.
  3. Open Incoming Webhooks → toggle On → Add New Webhook to Workspace → choose #nibble → Allow.
  4. Copy the webhook URL (it looks like https://hooks.slack.com/services/T000/B000/xxxx). Treat it like a password.

That URL is where a message gets posted. Because a static GitHub Pages site can't run scheduled jobs or keep a secret, the posting itself lives in the production version (section 7) — a small scheduled function that fires the same events the in-app toggles list:

  • Stall of the week — every Monday 9am, posts the featured stall.
  • New stall added — posts when an admin adds a stall.
  • Trending dish — posts the week's most-upvoted dish.

In Supabase this is a Scheduled Edge Function that reads the data and does a single POST to your webhook URL (stored as a secret). Rough shape of the message call:

await fetch(SLACK_WEBHOOK_URL, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    text: "🍜 Stall of the week: *Sawadee Thai Food Van* — go say hi!"
  })
});
What changes in the app

The Settings panel keeps the same three switches, relabelled from “Weekly email” to “Slack alerts”, so admins still choose which events post. Give me the webhook URL when you're ready and I'll wire the function up as part of the production build.

10Restrict the live site to @selinafinance.co.uk

For the real launch, only people with a Selina email should get in — using their work Google login (SSO), no passwords, no outsiders. This needs the backend from section 7 (a static site can't enforce logins). Once the app is on Supabase + Vercel/Netlify:

  1. In your Supabase project → Authentication → Providers → Google, toggle it on.
  2. In the Google Cloud Console, create an OAuth client (Web application), add your site URL and the Supabase callback URL it shows you, then paste the Client ID & Secret back into Supabase.
  3. Lock it to your domain so only Selina emails can sign in. Two layers, use both:
    • At sign-in: pass Google the hosted-domain hint hd=selinafinance.co.uk so the picker only offers Selina accounts.
    • On the server (the one that counts): a database rule / Auth Hook that rejects any account whose email doesn't end in @selinafinance.co.uk. This is what actually keeps outsiders out.
  4. Set admins (like you) in the database; everyone else signs in as an employee and can browse, review and upvote but not open the CMS.
One thing to decide

Your prototype currently uses selina.co for the demo login (Daisy Jones is daisy@selina.co). For the real SSO launch we'll switch the allowed domain to @selinafinance.co.uk — tell me if any other domains (contractors, a second brand) should also be let in.
